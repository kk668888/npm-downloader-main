# 安全审计说明

## 1. 文档目的

本文档用于说明当前项目的安全审计机制、接口约束、前后端交互流程、异常处理策略以及测试验证结果，作为后续开发、联调、上线评审和问题排查的参考依据。

当前实现目标不是“给出一个静态审计报告”，而是建立一条完整的安全控制链路：

- 依赖解析后必须先完成审计判断
- 审计异常不能默认放行
- 存在风险时必须显式确认后才能继续
- 确认动作必须绑定到当前任务，不能被伪造
- 等待确认不能无限挂起，必须具备超时清理能力

## 2. 审计流程

### 2.1 逻辑流程图

```text
发起任务（单包下载 / lockfile 上传）
  -> 服务端生成随机 taskId
  -> 服务端生成一次性 token
  -> 立即返回 taskId + token
  -> 后台开始解析依赖并执行安全审计
      -> 审计结果 = safe
          -> 直接继续下载处理
      -> 审计结果 = risky
          -> 任务进入 awaiting_audit_confirmation
          -> 前端展示审计报告
          -> 用户确认后继续处理
      -> 审计结果 = unavailable
          -> 任务进入 awaiting_audit_confirmation
          -> 前端提示“本次未完成有效审计”
          -> 用户确认后继续处理
      -> 用户长时间未确认
          -> 任务超时失败并清理等待状态
  -> 任务完成 / 失败
  -> 前端轮询任务状态并展示结果
```

### 2.2 核心设计原则

1. 审计不是旁路提示，而是下载流程的一部分。
2. 审计失败、超时、漏洞库不可用时，默认视为 `unavailable`，不能按“安全”处理。
3. 用户确认必须绑定具体任务，不能只依赖可猜测的任务编号。
4. 任务等待确认必须有超时回收，避免内存堆积和僵尸任务。

## 3. 审计状态模型

当前审计结果采用三态模型：

| 状态 | 含义 | 默认行为 |
| --- | --- | --- |
| `safe` | 已完成审计，未发现命中漏洞 | 自动继续执行 |
| `risky` | 已完成审计，发现命中漏洞 | 等待用户确认 |
| `unavailable` | 审计未成功完成，例如超时、接口异常 | 等待用户确认 |

这种建模的意义是把“确实安全”与“根本没审计成功”明确区分开，避免网络故障、registry 异常或第三方接口超时导致错误放行。

## 4. 后端实现说明

## 4.1 审计入口

后端支持两类任务：

- 单包下载：`POST /api/download-package`
- lockfile 上传：`POST /api/upload`

两条链路都会在任务创建时：

1. 生成随机 `taskId`
2. 为任务生成一次性 `token`
3. 先把 `taskId + token` 返回给前端
4. 后台继续执行依赖解析、漏洞审计和压缩下载流程

这样可以保证：

- 前端可以第一时间拿到任务上下文并开始轮询
- 审计进入等待确认时，前端已经具备继续操作所需凭证
- lockfile 链路不会因为服务端同步等待确认而导致请求卡死

## 4.2 漏洞数据来源

审计逻辑会把待检查依赖整理成 `name -> versions[]` 的结构，然后调用 npm advisory bulk 接口进行批量查询。

当前使用的是 npm 官方 advisory bulk API。

## 4.2.1 安全审计报告是如何生成的

当前安全审计报告不是人工录入，也不是前端本地拼装，而是由服务端基于依赖解析结果和 npm 官方漏洞数据动态生成。

完整生成过程如下：

1. 前端发起单包下载或 lockfile 上传任务。
2. 服务端解析用户输入，得到待审计依赖列表。
3. 服务端把依赖整理为 `包名 -> 版本集合` 的批量结构。
4. 服务端调用 npm 官方 advisory bulk API 拉取这些依赖对应的漏洞公告。
5. 服务端使用 `semver` 将本地依赖版本与漏洞公告中的 `vulnerable_versions` 规则逐一比对。
6. 服务端汇总命中的漏洞、受影响包、严重级别和异常原因，生成统一的 `AuditReport`。
7. 服务端把 `AuditReport` 挂到当前任务状态中。
8. 前端在轮询到任务进入审计确认阶段后，再通过审计报告接口读取并展示这份报告。

可以把它理解为“服务端实时生成，前端按任务读取”的模式。

### 生成来源拆解

安全审计报告主要由三部分数据组成：

| 数据来源 | 作用 | 说明 |
| --- | --- | --- |
| 用户提交的依赖信息 | 审计输入 | 来自单包名称/版本，或 lockfile 解析结果 |
| npm advisory bulk API | 漏洞数据源 | 提供官方漏洞公告与受影响版本范围 |
| 服务端审计逻辑 | 报告生成器 | 负责匹配版本、计算风险状态、组装最终报告 |

### 报告生成后的消费方式

报告生成完成后，并不会直接写成静态文件，而是作为任务运行时数据被消费：

- 如果结果为 `safe`，任务继续执行，前端通常不会弹出审计确认
- 如果结果为 `risky`，前端展示漏洞命中报告，等待用户确认
- 如果结果为 `unavailable`，前端展示“本次未完成有效审计”的报告，等待用户确认

对应前端读取方式为：

```http
GET /api/task/:taskId/audit
```

因此，当前“安全审计报告”的本质是：**服务端围绕当前任务实时生成的一份结构化审计结果**，而不是独立的离线文档产物。

## 4.3 版本命中判断

版本匹配基于标准 `semver` 能力进行判断，而不是手写字符串比较。当前处理原则如下：

1. 尝试把依赖版本规范化为可比较的 semver 版本。
2. 使用漏洞规则中的 `vulnerable_versions` 范围进行命中判断。
3. 如果版本无法规范化，出于保守策略，按“可能命中”处理。

该策略优先保证安全性，避免因为版本解析失败而误放行风险依赖。

## 4.4 超时与异常策略

审计存在以下异常场景：

- npm advisory 接口超时
- npm advisory 接口返回错误
- 第三方返回结构异常
- 网络暂时不可用

当前策略统一为：

- 不把异常降级为 `safe`
- 返回 `auditStatus = unavailable`
- 给出 `reason` 字段描述异常原因
- 任务进入“等待审计确认”状态

这项策略修复了早期“审计失败自动继续下载”的核心缺陷。

## 5. 审计确认机制

## 5.1 任务标识

每个任务包含两类标识：

- `taskId`：任务公开标识，用于轮询状态、查看报告
- `token`：任务私有确认凭证，用于执行确认操作

其中：

- `taskId` 使用随机值生成，避免使用时间戳等可预测方案
- `token` 只在任务创建响应中返回，不通过公共状态接口暴露

## 5.2 为什么不能只靠 taskId

如果只有 `taskId` 而没有附加校验，攻击者只要猜到任务编号，或能读取任务状态，就可能替别的请求执行“继续下载”确认。

因此当前确认接口必须同时满足：

1. `taskId` 存在
2. `token` 正确
3. 当前任务确实处于等待确认状态

否则接口直接拒绝。

## 5.3 确认接口

确认接口：

```http
POST /api/task/:taskId/confirm-audit
Content-Type: application/json

{
  "token": "..."
}
```

服务端校验规则：

- `token` 无效：返回 `403`
- 当前任务并不处于待确认状态：返回 `409`
- 校验通过：返回 `200`，并唤醒等待中的任务

这种处理方式可以避免“前端看起来确认成功，服务端实际没继续”的假成功问题。

## 6. 状态查询与暴露控制

任务状态查询接口：

```http
GET /api/task/:taskId
```

该接口只返回任务公共状态，不返回 `token`。这样可以支持前端轮询和状态展示，同时避免确认凭证从状态接口泄露。

当前状态信息主要用于：

- 展示下载进度
- 判断任务是否进入审计确认阶段
- 恢复审计报告弹窗
- 获取最终下载结果或失败原因

## 7. 等待确认与超时清理

当审计结果为 `risky` 或 `unavailable` 时，任务会进入等待确认状态。

为了避免任务无限挂起，服务端实现了确认超时机制：

1. 进入等待状态时注册确认句柄
2. 同时启动超时计时器
3. 用户确认后，立即释放等待句柄并清理定时器
4. 超时未确认时，任务自动失败并清理内部状态

这部分主要解决两个问题：

- 防止内存中的等待 Promise 永久堆积
- 防止用户关闭页面后任务长期处于不可回收状态

## 8. 前端交互说明

## 8.1 当前交互模型

当前产品约束为：**同一时刻只允许一个任务进行**。

因此前端采用单任务交互模型：

- 任务创建后缓存当前 `taskId` 对应的 `token`
- 轮询任务状态
- 如果状态进入等待审计确认，则展示审计报告弹窗
- 用户确认后调用确认接口，并恢复轮询

在这个前提下，前端不需要支持多任务并发的审计弹窗管理。

## 8.2 审计报告展示

当前前端在以下场景展示审计报告：

- 审计命中漏洞
- 审计不可用，需要用户确认是否继续
- 用户刷新后重新进入页面，通过任务状态恢复弹窗

前端行为要求如下：

1. 只有确认接口调用成功，才允许关闭等待状态
2. 如果确认接口失败，必须保留当前审计上下文并提示错误
3. 前端可以重新打开当前任务的审计报告

## 9. CORS 与接口安全边界

服务端当前采用显式白名单 CORS 策略：

- 只允许配置中的前端来源访问 API
- 禁用框架默认的宽松 CORS 行为

这样做的目的有两个：

1. 减少第三方页面直接调用任务相关接口的可能性
2. 保证跨域策略在应用层只有一处生效，避免“看起来收紧了，实际被默认配置放开”的情况

需要注意的是，CORS 只是浏览器层面的限制，不应替代鉴权和任务凭证校验。真正决定能否确认任务的仍然是 `taskId + token + 当前任务状态`。

## 12. 本轮修复覆盖的问题

本轮主要修复了以下历史问题：

1. 审计超时或异常时被错误视为“安全”
2. `taskId` 可预测，确认接口可被伪造调用
3. `token` 可从状态接口读取，导致保护失效
4. lockfile 上传链路同步等待确认，导致前端无法拿到任务上下文
5. 审计等待状态没有超时回收，可能造成任务长期挂起
6. 确认接口失败时，前端和服务端状态可能不一致
7. 版本命中规则过于粗糙，审计结果可信度不足
8. 默认 CORS 配置可能放宽审计接口暴露范围

## 13. 当前边界与后续建议

当前实现已经满足现阶段“单任务、显式确认、失败不放行”的目标，但仍有明确边界：

1. 当前前端模型基于“同一时刻只允许一个任务”设计。
2. 如果未来产品要支持多任务并发，前端需要改成按 `taskId` 独立维护审计弹窗、轮询函数和任务上下文。
3. 如果未来需要更强的权限控制，可以继续在 `taskId + token` 之外叠加用户身份、会话或签名机制。

## 14. 结论

从本轮审计结果看，当前项目的安全审计链路已经从“提示型能力”升级为“流程型控制能力”。

其核心改进体现在：

- 审计失败不再默认放行
- 风险确认必须绑定具体任务
- 等待确认具备超时和清理能力
- lockfile 与单包下载链路都能完成完整审计闭环
- 前后端状态交互已经具备可恢复、可拒绝、可追踪的基本能力

在当前“单任务”产品约束下，这套方案已经可以作为上线说明和后续维护基线使用。

### 🚀 优化建议

- 如果后续进入多用户或更高安全等级场景，建议把 `token` 升级为带有效期的签名凭证，并绑定用户上下文。
- 可以为审计链路补充结构化埋点，例如审计超时率、用户确认率、`unavailable` 占比和高风险包命中率。
- 可以在前端增加“任务互斥中”提示，进一步强化“单任务模型”的产品约束，减少误操作。
